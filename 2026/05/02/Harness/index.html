Harness
2026-05-02 16:36:06
  • Harness design for long-running application development

Overview

Harness 工程的本质，不是让模型本身变得更聪明，而是通过外部结构把模型的输出变得可约束、可检查、可迭代。

Anthropic 的文章给了一个很清晰的结构：Planner, Generator, Evaluator。Planner 把一句话需求扩展成产品 spec；Generator 按模块实现；Evaluator 使用真实工具测试结果。真正关键的连接层是 Contract：在每个 sprint 开始前，Generator 和 Evaluator 先对「这轮到底算完成什么」达成一致，把模糊目标转成可验证的 checklist。

也就是说，Harness 的核心不是多 Agent，而是把模型容易失败的地方显式工程化：先识别失败模式，再为每种失败模式设计外部约束、工具和反馈回路。

Contract Example

一个典型的 Contract 不应该只描述 UI 上有什么，而应该描述用户能力、完成标准和可复现的 QA 路径。

Evaluator 的价值在于，它不会只看「页面上有没有 Delete 按钮」，而是实际执行用户路径。

更理想的 Evaluator 输出应该是可复现的 bug report，而不是一句「不符合预期」。

这就是 Contract 的意义：它不是写给人看的流程文档，而是给 Evaluator 提供一个可以执行、可以判定、可以回归的测试面。

Design Pattern of AI Agent

这里要注意，Evaluator 本身也不是天然可靠的。Anthropic 在文章里提到，Claude 默认做 QA 时也会偏宽松：它可能发现了真实问题，但又说服自己「这不算大问题」，最后仍然通过。因此，Evaluator 的 prompt、评分标准和 few-shot examples 也需要持续调试。

换句话说，Harness 不是把一个模型拆成几个名字不同的模型，而是让不同角色承担不同的压力：Generator 负责实现，Evaluator 负责怀疑，Contract 负责把争论限制在可验证的范围里。

Boundary

Harness 也不是免费午餐。

Anthropic 的例子里，solo run 大约 20 分钟、9 美元；full harness 大约 6 小时、200 美元。新版 DAW harness 也接近 4 小时、124.70 美元。它带来的质量提升很明显，但成本、延迟和编排复杂度也显著增加。

Summary

随着模型能力变强，这些辅助式的架构或工具看起来似乎会变得不再重要，因为模型本身已经可以应对足够复杂的长时间任务。

但我的理解恰好相反：

具体来说，Skill、MCP、Harness 都是在处理模型能力边界，但它们处理的是不同层次的问题：

设计一个系统，往往不是把所有东西都内化成一个通用范式，而是持续观察当前基础模型的能力边界在哪里，然后把边界处的失败模式工程化。

新的模型发布，会解决一部分旧边界，也会暴露新的边界。Agent 工程真正有价值的地方，就是不断找到这些新的边界，并为它们设计合适的 Harness。
